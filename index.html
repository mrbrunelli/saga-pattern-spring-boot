Saga Pattern
orquestração no Spring Boot

Como manter consistência quando uma operação de negócio atravessa vários serviços que não compartilham a mesma transação ACID — e como desfazer tudo, em ordem, quando algo dá errado.

01 — O problema

Por que uma transação só não resolve

Imagine finalizar um pedido de e-commerce. São quatro efeitos que precisam acontecer juntos: criar o pedido, cobrar o pagamento, reservar o estoque e agendar o envio. O detalhe é que esses efeitos vivem em contextos diferentes — bancos separados, às vezes microsserviços distintos. Não existe um @Transactional que envolva todos eles de uma vez.

O cenário perigoso: você já cobrou o cliente e, no passo seguinte, descobre que o estoque acabou. Sem um mecanismo de rollback, o dinheiro saiu e o produto não existe. A Saga resolve isso dando a cada passo um par: uma ação e uma compensação (a operação inversa). Falhou no meio? Roda as compensações dos passos já concluídos, na ordem inversa.

02 — Duas abordagens

Orquestração vs. coreografia

A Saga tem dois sabores. Aqui o foco é a orquestração, pela clareza do fluxo e pela facilidade de raciocinar sobre o rollback.

foco aqui

Orquestração

Um maestro central

Um componente decide a ordem dos passos e dispara as compensações. Fluxo explícito, fácil de debugar e de visualizar. Ideal quando há lógica de rollback de verdade.

Coreografia

Cada um reage

Sem maestro: cada serviço escuta eventos e reage publicando os seus. Mais desacoplado, porém o fluxo fica espalhado — difícil de seguir e de depurar conforme cresce.

04 — Anatomia

O contrato e o maestro

Todo passo conhece duas coisas: como se executar e como se compensar. Esse contrato simples é o que torna o orquestrador genérico.

// SagaStep.kt — o contrato
interface SagaStep<C> {
    val name: String
    fun execute(context: C)
    fun compensate(context: C)   // operação inversa, idempotente
}

O orquestrador empilha cada passo bem-sucedido. Se um execute() lança exceção, ele desempilha e compensa na ordem inversa (LIFO) — exatamente como na simulação acima.

// SagaOrchestrator.kt — o maestro
@Component
class SagaOrchestrator {
    fun <C> run(context: C, steps: List<SagaStep<C>>) {
        val executed = ArrayDeque<SagaStep<C>>()
        try {
            for (step in steps) {
                step.execute(context)
                executed.addLast(step)        // só empilha se teve sucesso
            }
        } catch (ex: Exception) {
            while (executed.isNotEmpty()) {    // compensa em ordem inversa
                val step = executed.removeLast()
                runCatching { step.compensate(context) }
                    .onFailure { /* alerta + revisão manual */ }
            }
            throw SagaExecutionException("Saga abortada e compensada", ex)
        }
    }
}

Cada passo concreto adapta um serviço participante. Repare como o execute grava o ID gerado no contexto e o compensate o lê — assim a compensação só desfaz o que de fato aconteceu.

// PaymentStep.kt — um passo concreto
@Component
class PaymentStep(private val payments: PaymentService) : SagaStep<OrderSagaContext> {
    override val name = "PROCESSAR_PAGAMENTO"

    override fun execute(context: OrderSagaContext) {
        context.paymentId = payments.charge(context.customerId, context.amount)
    }

    override fun compensate(context: OrderSagaContext) {
        context.paymentId?.let { payments.refund(it) }   // estorno idempotente
    }
}
05 — Próximo nível

Evoluindo para eventos assíncronos

A versão acima é síncrona — perfeita para entender o padrão e escrever testes. A orquestração de eventos "de produção" costuma ser assíncrona, e é aí que entra a mensageria (SQS, por exemplo).

O orquestrador deixa de chamar métodos direto e passa a publicar comandos em filas por serviço (payment-commands, inventory-commands), enquanto escuta uma fila de respostas. A mudança crítica: o estado da saga vira persistente — você grava numa tabela qual passo está pendente, qual concluiu, qual compensou. Assim a saga sobrevive a um restart de pod no meio do fluxo, em vez de deixar o pedido num limbo.

06 — Coreografia na prática

Cada serviço reage aos seus eventos

Na coreografia não há maestro. O OrderService dispara um OrderCreatedEvent e o fluxo segue por reação em cadeia: cada serviço escuta os eventos que lhe interessam e publica o próximo quando conclui — ou um evento de falha quando não consegue. A compensação também é orientada a eventos: quando o estoque falha, ele publica StockReservationFailedEvent e o serviço de pagamento ouve e estorna sozinho.

Os eventos como contratos entre serviços

Cada serviço só conhece os eventos que produz ou consome — nunca chama o outro diretamente. Com Spring, ApplicationEventPublisher cobre o caso síncrono em processo; para sistemas distribuídos, troca-se por Kafka ou SQS mantendo a mesma semântica.

// Events.kt — contratos de domínio (sem acoplamento entre serviços)
data class OrderCreatedEvent(val orderId: String, val customerId: String, val amount: BigDecimal)
data class PaymentProcessedEvent(val orderId: String, val paymentId: String)
data class StockReservationFailedEvent(val orderId: String, val reason: String)
data class StockReservedEvent(val orderId: String, val reservationId: String)
// PaymentEventHandler.kt — reage e compensa de forma autônoma
@Component
class PaymentEventHandler(
    private val payments: PaymentService,
    private val events: ApplicationEventPublisher
) {
    @EventListener
    fun onOrderCreated(e: OrderCreatedEvent) {
        val paymentId = payments.charge(e.customerId, e.amount)
        events.publishEvent(PaymentProcessedEvent(e.orderId, paymentId))
    }

    @EventListener                              // compensação autônoma
    fun onStockFailed(e: StockReservationFailedEvent) {
        payments.refund(e.orderId)              // estorno idempotente
    }
}
// InventoryEventHandler.kt — publica falha em vez de lançar exceção
@Component
class InventoryEventHandler(
    private val inventory: InventoryService,
    private val events: ApplicationEventPublisher
) {
    @EventListener
    fun onPaymentProcessed(e: PaymentProcessedEvent) {
        runCatching { inventory.reserve(e.orderId) }
            .onSuccess { id -> events.publishEvent(StockReservedEvent(e.orderId, id)) }
            .onFailure { ex -> events.publishEvent(StockReservationFailedEvent(e.orderId, ex.message ?: "sem estoque")) }
    }
}
07 — Checklist de produção

O que separa o exemplo do real

▪ 01

Idempotência

A mesma mensagem pode chegar duas vezes na fila. Cada handler — e cada compensação — precisa detectar reprocessamento e não duplicar o efeito.

▪ 02

Estado persistente

Grave o progresso da saga em banco (com auditing de createdAt/updatedAt). Sem isso, um deploy no meio do fluxo perde o rastro do que falta compensar.

▪ 03

DLQ + backoff

Distinga falha transitória (retry com backoff exponencial) de falha de negócio como "estoque insuficiente" (dispara compensação, sem retry).

▪ 04

Compensação que falha

É o pior caso: não pode bloquear as demais. Registre, alerte e mande para revisão manual — nunca relance interrompendo o rollback.

Roteiro de prática

① Rode o exemplo síncrono  →  ② Escreva um teste verificando que refund é chamado quando o estoque falha  →  ③ Quebre em filas SQS adicionando a entidade JPA de estado.
